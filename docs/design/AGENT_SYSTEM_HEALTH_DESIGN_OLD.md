# Agent 系统健康模块 - 完整实现设计

## 文档说明

本文档提供完整的、可落地的实现设计，包括：
- 与现有架构的集成方案
- 详细的接口定义（Go 代码）
- 数据流和调用链路
- 分阶段实施计划
- 具体的代码改动位置

## 1. 背景与目标

当前系统已经具备 AI Chat、TwoPhaseAgent、工具调用、后台任务和 Git 命令执行能力，但运行期的错误处理仍然是分散的：

- 错误来源多，缺少统一采集入口
- 错误上下文不完整，开发复盘成本高
- 运行时问题往往只有原始报错，没有根因分析
- Agent 具备软件结构认知能力，但没有稳定接入运行期健康诊断链路
- 现有日志更偏原始输出，不够面向修复工作流

本设计目标不是再加一个“日志模块”，而是建立一套完整的 **Agent 系统健康平台**，让系统在运行过程中能够：

1. 持续采集错误、异常、关键风险事件与运行状态。
2. 基于统一事件模型记录问题，并保存足够的上下文。
3. 由 Agent 结合架构知识、实时数据和历史轨迹分析问题。
4. 输出开发可直接消费的诊断内容，支持后续修复和优化。
5. 在 UI、文件持久化、开发排障流程中形成闭环。

## 2. 非目标

以下内容不属于本设计的一期强制范围：

- 自动修复生产问题
- 自动执行恢复动作
- 将健康数据上报到外部云服务
- 替代现有日志系统的全部职责
- 引入新的第三方监控平台依赖

本设计强调“诊断增强”和“可观测性增强”，不是全自动自治系统。

## 3. 设计原则

### 3.1 统一入口

所有健康事件必须通过统一采集接口进入系统，不能继续散落在各模块各自打印和各自处理。

### 3.2 上下文优先

没有上下文的错误记录价值有限。每条健康事件都必须尽可能携带可诊断信息，而不是仅记录一条字符串。

### 3.3 Agent 参与，但不裸奔

Agent 不能直接读取一切原始运行态杂讯。必须先经过标准化、脱敏、裁剪、聚合，再生成可分析的上下文包。

### 3.4 机器可处理 + 人可阅读

输出必须同时满足两类消费：

- 机器可检索、可聚合、可去重
- 开发可直接阅读、复盘、定位、修复

### 3.5 低侵入、可渐进落地

健康模块需要覆盖整个系统，但接入方式必须支持分阶段推进，避免一次性大规模侵入所有模块。

### 3.6 安全与脱敏优先

健康记录绝不能把 API Key、Token、密码、敏感仓库内容、用户隐私输入直接落盘或暴露给 Agent。

## 4. 问题定义

“系统健康”在这里不是单指 crash，而是覆盖以下四类信息：

1. **故障事件**
   - panic
   - error 返回
   - 工具调用失败
   - Git 命令失败
   - IO/配置/解析失败

2. **风险事件**
   - 重试次数异常
   - 同类错误短时间连续出现
   - Agent 规划失败、执行中断、等待确认超时
   - 后台任务长时间无响应

3. **运行态状态**
   - 当前 UI context
   - 当前 repo 状态摘要
   - 当前 Agent phase / plan / step
   - 最近操作链路

4. **诊断产物**
   - 根因猜测
   - 影响判断
   - 修复建议
   - 相似历史事件

## 6. 健康事件模型

### 6.1 统一事件结构

建议定义统一的 `HealthEvent`：

```yaml
event_id: "evt_20260306_000001"
timestamp: "2026-03-06T14:21:35+08:00"
severity: "error"
category: "agent_execution"
source_module: "pkg/ai/agent/two_phase_agent"
source_function: "execute"
error_code: "AGENT_TOOL_EXEC_FAILED"
summary: "关键步骤执行失败"
raw_error: "未知工具: add"
stack_trace: ""
message: "关键步骤失败: 暂存当前修改 — 未知工具: add"
run_id: "run_20260306_a1"
session_id: "chat_20260306_01"
repo_context:
  repo_root: "masked"
  branch: "main"
  worktree_state: "dirty"
runtime_context:
  os: "windows"
  app_version: "local-dev"
  current_view: "aiChat"
agent_context:
  phase: "executing"
  current_step_id: "2"
  current_step_desc: "暂存当前修改"
user_context:
  latest_action: "AI Chat: 帮我提交当前修改"
command_context:
  tool_name: "git_add"
  command_summary: "git add ..."
payload:
  sanitized_params:
    files: ["masked"]
tags:
  - "critical"
  - "agent"
  - "tool"
```

### 6.2 事件等级

定义四级严重度：

- `info`
  - 普通运行事件、状态切换、成功恢复
- `warning`
  - 可继续运行但已有异常征兆
- `error`
  - 功能失败，需要记录和分析
- `critical`
  - panic、连续失败、关键链路中断、严重数据损坏风险

### 6.3 事件分类

建议分类至少包括：

- `gui_error`
- `agent_plan`
- `agent_execution`
- `agent_state`
- `git_command`
- `os_command`
- `background_task`
- `config`
- `persistence`
- `panic`
- `performance`
- `health_internal`

## 7. 事件来源与采集策略

### 7.1 采集范围

健康模块必须覆盖以下入口：

1. GUI 层
   - 弹窗异常
   - 视图渲染失败
   - context 切换失败

2. Agent 层
   - 规划失败
   - 等待确认卡死
   - 执行步骤失败
   - 工具不存在
   - 工具返回失败

3. Commands 层
   - Git 命令执行错误
   - 外部命令启动失败
   - 命令超时

4. 后台任务层
   - worker panic
   - 任务超时
   - 队列积压

5. 配置与持久化层
   - 配置解析失败
   - 文件读写失败
   - 状态持久化失败

6. 全局异常
   - panic/recover
   - 未捕获 error 汇聚

### 7.2 采集时机

按三种方式采集：

1. **主动上报**
   - 业务逻辑发现错误时显式上报

2. **包装拦截**
   - 对关键执行器统一包装，例如 Agent 工具执行器、命令执行器、任务执行器

3. **全局恢复**
   - 对 goroutine、主循环、关键 worker 加 recover 兜底

### 7.3 事件去重

为避免错误风暴，需要引入去重与合并策略：

- 同一模块、同一错误码、同一摘要在短时间窗口内合并
- 保留首发时间、最近发生时间、发生次数
- 对 critical 事件不去重，只聚类显示

## 8. 上下文构建设计

Agent 分析的质量取决于喂给它的上下文包。上下文必须是“诊断视角”的，不是原始日志堆砌。

### 8.1 上下文包组成

建议 `HealthAnalysisContext` 至少包含：

- 事件基础信息
- 最近 N 条相关健康事件
- 当前 UI context 与焦点视图
- 最近操作轨迹
- 当前 repo 摘要
- 当前 Agent 状态
- 最近工具调用链
- 当前配置摘要
- 相似历史问题摘要
- 设计文档引用摘要

### 8.2 Repo 上下文

建议纳入：

- 当前分支
- staged / unstaged 文件数量
- 最近 5 条提交摘要
- 是否 rebase / merge / cherry-pick 中
- worktree / submodule 状态摘要

### 8.3 Agent 上下文

建议纳入：

- 当前 phase
- 当前 plan summary
- 当前 step
- 最近 step update
- 最近 tool call / tool result
- 最近一次用户指令摘要

### 8.4 操作轨迹

维护最近操作环形缓冲区，例如：

- 进入 AI Chat
- 用户输入指令
- Agent 开始规划
- 调用只读工具
- 生成执行计划
- 用户确认
- 开始执行步骤 2
- `git add` 失败

### 8.5 设计知识接入

Agent 不应只看运行时数据，还应熟悉软件设计。这里建议接入：

- `docs/ARCHITECTURE.md`
- `docs/PROJECT_OVERVIEW.md`
- `docs/design/AI_CHAT_DESIGN_V2.md`
- 与错误模块直接相关的设计文档

注意：

- 不做全量文档灌输
- 只按模块和错误类别选取相关文档摘要
- 文档接入由 `HealthContextBuilder` 做裁剪

## 9. 脱敏与安全设计

### 9.1 必须脱敏内容

以下内容禁止直接写入健康记录或传给 Agent：

- API Key
- Token
- Password / Passphrase
- 绝对敏感路径
- 用户完整自由输入原文
- 命令中的秘密参数
- 可能包含私密代码的完整 diff

### 9.2 脱敏策略

- 保留字段类型，不保留敏感值
- 对路径做前缀折叠或掩码
- 对长文本做摘要化
- 对命令行只保留工具名和参数结构，不保留敏感值
- 对用户输入保留意图摘要，不保留原始全文

### 9.3 Agent 输入边界

Agent 分析用上下文包只能来自：

- 脱敏后的结构化事件
- 摘要化状态快照
- 选定文档摘要

禁止让 Agent 直接扫描所有原始日志和完整本地文件。

## 10. 存储设计

### 10.1 双层存储

#### 内存层

用于实时 UI 与分析：

- 最近 500 条事件
- 最近 100 个 error/critical 事件索引
- 最近 50 个分析结果
- 最近操作轨迹缓冲区

#### 磁盘层

用于持久化和复盘：

- 原始事件流：`JSONL`
- 分析结果：`Markdown + JSON`
- 周期汇总：可选

### 10.2 文件组织建议

```text
health/
├── events/
│   ├── 2026-03-06.jsonl
│   └── 2026-03-07.jsonl
├── analyses/
│   ├── evt_20260306_000001.md
│   ├── evt_20260306_000001.json
│   └── ...
└── indexes/
    └── latest_summary.json
```

### 10.3 保留策略

- 原始事件流按天滚动
- 分析结果按事件保存
- 超过容量的历史文件按策略清理
- 关键事件可单独保留更长时间

## 11. Agent 分析策略

### 11.1 分析分级

不是每条事件都做同样深度的分析，建议分三级：

#### A. 轻分析

触发条件：

- warning / error 事件

输出：

- 分类
- 是否重复
- 是否建议升级
- 简短原因猜测

#### B. 深分析

触发条件：

- critical 事件
- panic
- 同类错误短时间高频发生
- 关键链路连续失败
- 开发手动触发

输出：

- 问题摘要
- 根因猜测
- 证据链
- 影响范围
- 复现线索
- 修复建议
- 置信度
- 待人工确认项

#### C. 聚类与趋势分析

触发条件：

- 同类问题达到阈值
- 某版本后错误显著增加

输出：

- 相似事件归并
- 疑似回归
- 高频失败链路
- 稳定性趋势

### 11.2 分析输出结构

建议统一输出 `HealthAnalysisReport`：

```yaml
report_id: "rpt_evt_20260306_000001"
event_id: "evt_20260306_000001"
analysis_level: "deep"
summary: "Agent 执行阶段调用了不存在的工具别名，导致关键步骤失败。"
root_causes:
  - "计划生成阶段与执行阶段工具命名不一致"
evidence:
  - "当前步骤 tool_name=add"
  - "执行注册表中不存在 add，仅存在 git_add"
impact:
  - "AI Chat 提交代码流程中断"
reproduction:
  - "在 AI Chat 中输入：帮我提交当前修改"
fix_suggestions:
  - "增加工具别名映射"
  - "在计划校验阶段提前验证 tool_name"
confidence: 0.87
manual_checks:
  - "确认是否还有其他历史别名未覆盖"
related_docs:
  - "docs/design/AI_CHAT_DESIGN_V2.md"
```

## 12. 健康策略引擎

`HealthPolicyEngine` 负责决定：

- 事件是否需要落盘
- 是否需要轻分析
- 是否需要深分析
- 是否应升级 severity
- 是否生成 UI 提示
- 是否参与聚类

### 12.1 规则示例

1. `panic` 必须：
   - 立即记录
   - 立即深分析
   - UI 高优先提示

2. `Agent PhaseExecuting + Critical StepFailed` 必须：
   - 升级为 critical
   - 立即深分析

3. 同一错误 5 分钟内出现 3 次：
   - 生成聚类事件
   - 触发趋势提示

4. 一般 warning：
   - 记录
   - 延迟轻分析或仅聚合

## 14. 开发工作流设计

健康模块的价值在于帮助开发修复问题，所以输出必须能直接进入开发流程。

### 14.1 典型工作流

1. 运行中发生错误
2. 健康模块记录事件并归档上下文
3. 策略引擎决定触发深分析
4. Agent 生成诊断报告
5. 开发在健康中心或磁盘报告中查看：
   - 发生位置
   - 复现线索
   - 根因猜测
   - 修复建议
6. 开发修复后基于相同事件模型补测试与回归验证

### 14.2 面向开发的报告标准

每份报告至少包含：

- 事件编号
- 发生时间
- 影响模块
- 严重等级
- 问题摘要
- 根因猜测
- 证据链
- 修复建议
- 验证建议

## 15. 性能与稳定性要求

### 15.1 性能要求

- 健康事件采集不能阻塞主交互线程
- 事件标准化应尽量轻量
- 深分析异步执行
- 高并发错误场景下要有限流与合并

### 15.2 稳定性要求

- 健康模块本身不能成为新的崩溃源
- HealthAnalyzer 失败时不能影响主流程
- 存储失败要降级但不能吞掉主错误
- 健康模块内部错误需要自监控，归类为 `health_internal`

## 17. 风险与应对

### 风险 1：噪音太多

问题：

- 健康事件数量大，开发看不过来

应对：

- 分级
- 去重
- 聚类
- 规则筛选

### 风险 2：上下文过重

问题：

- 采集太多内容，影响性能和分析质量

应对：

- 摘要化
- 限流
- 上下文裁剪
- 仅在深分析时扩展上下文

### 风险 3：敏感信息泄露

问题：

- 日志或 Agent 输入泄露敏感内容

应对：

- 统一脱敏器
- 白名单字段
- 报告生成前二次脱敏

### 风险 4：Agent 误判

问题：

- 分析结果可能不准确

应对：

- 输出证据链与置信度
- 明确“待人工确认项”
- 不允许 Agent 分析结果直接自动修复

## 18. 验收标准

以下标准满足后，才算健康模块设计达到可实施状态：

1. 系统内主要错误入口都能统一映射到 `HealthEvent`。
2. 每条 `error/critical` 事件都能关联到最小可用上下文包。
3. `critical` 事件支持自动触发 Agent 深分析。
4. Agent 分析结果输出结构固定，至少包含摘要、根因、证据、影响、修复建议、置信度。
5. 健康数据支持内存查询和磁盘持久化。
6. 敏感信息脱敏规则明确且可验证。
7. UI 中存在明确的健康入口和错误详情查看能力。
8. 健康模块内部失败不会拖垮主业务流程。

## 19. 对当前项目的落地建议

结合当前仓库结构，建议未来落地时优先接入以下模块：

- `pkg/ai/agent/`

- `pkg/commands/git_commands/`
- `pkg/commands/oscommands/`
- `pkg/tasks/`
- `pkg/config/`
- `pkg/gui/`

同时优先复用以下已有能力：

- Agent phase / step 状态
- repo context 构建逻辑
- AI Chat 现有上下文与诊断能力
- 现有 docs 中的架构与 AI 设计文档

## 20. 结论

Agent 系统健康模块的关键价值，不在于“又多记了一份日志”，而在于把运行时错误转化为开发可执行的诊断资产。

这套模块建成后，系统将具备以下能力：

- 知道哪里出了错
- 知道错发生时系统在做什么
- 知道错之前发生了什么
- 知道错大概率为什么发生
- 知道开发下一步该怎么修

这才是一个真正有工程价值的健康平台，而不是表面上“接了 AI”的报错包装器。
